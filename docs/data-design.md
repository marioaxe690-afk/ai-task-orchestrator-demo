# Data Design

这个仓库的核心不是数学内容，而是任务识别与任务拆解数据。

## 设计目标

Demo 要展示 AI 如何处理混乱输入：

```text
用户上传截图或输入混乱信息
-> AI 识别真实需求
-> 判断简单元任务还是复杂组合任务
-> 必要时追问
-> 拆成可执行元任务
-> 标注难度、耗时、完成标准和失败风险
-> 根据用户反馈调整下一步
```

## 为什么不用题库结构

题库通常围绕题干、答案、解析、知识点和难度建立索引。这个项目不这样做，因为演示重点不是“存多少道题”，而是“系统如何决定下一步该做什么”。

同一张数学截图可能对应完全不同的编排结果：

- 用户说“这题怎么做”：进入截图理解和题目解析。
- 用户说“明天考试这个还没看懂”：进入临考救急和范围压缩。
- 用户只说“这个不会”：先进入澄清目标。
- 用户说“计划太满”：进入反馈调整和任务压缩。

因此主数据必须拆成输入案例、意图标签、元任务、组合模板、反馈闭环和 Demo 场景，而不是普通题目列表。

## 数据边界

允许：

- 模拟用户输入
- 模拟截图素材路径
- 静态任务拆解数据
- 静态反馈闭环
- 前端 Demo 组装示例

不允许：

- 真实隐私截图
- API Key 或真实账号数据
- 真实 AI API 接入
- 后端服务或数据库依赖
- 把数据扩展成普通数学题库
